================================================================================
                              ROLLBACK PROCEDURE
================================================================================

If critical issues occur during or after deployment:

OPTION 1: RAILWAY UI ROLLBACK (Easiest)
════════════════════════════════════════
1. Go to Railway dashboard → Card-Benefits project
2. Click "Deployments" tab
3. Find the previous successful deployment
4. Click the "..." menu → "Rollback to this deployment"
5. Confirm the rollback
6. Wait 2-3 minutes for restart

Effect: Reverts to previous code AND database schema
Time to restore: ~3-5 minutes
Risk: LOW (reverting to last known good state)

OPTION 2: GIT REVERT (If code is the issue)
═════════════════════════════════════════════
If you want to revert just the code:

  git revert HEAD
  git push origin main

Railway will automatically redeploy with reverted code.
Effect: Reverts code only (database keeps new schema)
Time: ~10-15 minutes (includes full build)
Risk: MEDIUM (database schema may be incompatible)

OPTION 3: MANUAL DATABASE ROLLBACK
═══════════════════════════════════
If only database is problematic:

  npx prisma migrate resolve --rolled-back 20260403_add_value_history_tracking
  npx prisma migrate resolve --rolled-back 20260403100000_add_admin_feature_phase1
  # ... and so on for others

Effect: Reverts migrations only
Time: < 2 minutes
Risk: MEDIUM (may lose data if new fields were populated)

WHEN TO USE EACH:
─────────────────
- Application crash: Use Option 1 (full rollback)
- Migrations failed: Use Option 1 (full rollback)
- Performance issues: Monitor for 1 hour, then decide
- Bug in code: Use Option 2 (code revert)
- Database corruption: Contact Railway support
