TECHNICAL DECISIONS
================================================================================

1. PAGINATION PATTERN
   Decision: Page-based (1-indexed) pagination
   Reason: Simple, intuitive for UI, standard in web APIs
   Reference: /api/cards/available (working example)

2. LIMITS
   Master cards: default=12, max=50
   User cards: default=20, max=100
   Reason: User cards have nested data, higher limit appropriate
   Security: Prevents DoS with maximum enforcement

3. SUMMARY CALCULATION
   User cards: Summary from ALL cards, not paginated subset
   Reason: Dashboard needs accurate totals regardless of current page
   Impact: Users see correct wallet statistics always

4. PARALLEL QUERIES
   Decision: Use Promise.all() for count + paginated results
   Reason: More efficient than sequential queries
   Performance: Reduces database round-trips

5. RESPONSE STRUCTURE
   Decision: Add pagination field (new schema)
   Reason: More semantic, includes all pagination info
   Migration: Breaking change, but properly documented
