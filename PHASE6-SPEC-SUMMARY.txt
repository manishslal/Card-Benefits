6 API ENDPOINTS TO CREATE
================================================================================

1. PATCH /api/cards/[id] - Edit Card
   - Update: customName, actualAnnualFee, renewalDate
   - Optimistic locking (version field)
   - Returns 200 OK or 409 Conflict (version mismatch)

2. DELETE /api/cards/[id] - Delete Card
   - Soft delete (status = 'DELETED')
   - Cascade: Archives all benefits
   - Requires confirmation string (card name)
   - Returns 200 OK with count of deleted benefits

3. POST /api/benefits/add - Add Benefit
   - Add from master catalog OR create custom
   - Unique constraint: (userCardId, name)
   - Returns 201 Created or 409 Conflict (duplicate)

4. PATCH /api/benefits/[id] - Edit Benefit
   - Update: name, userDeclaredValue, resetCadence, expirationDate
   - Optimistic locking (version field)
   - Returns 200 OK or 409 Conflict

5. DELETE /api/benefits/[id] - Delete Benefit
   - Soft delete (status = 'ARCHIVED')
   - Returns 200 OK

6. PATCH /api/benefits/[id]/toggle-used - Mark Used (Improve Existing)
   - Toggle isUsed flag
   - Increment timesUsed when marking used
   - Returns 200 OK

KEY ARCHITECTURE PATTERNS
================================================================================

OPTIMISTIC LOCKING (Race Condition Prevention):
- Every card/benefit has version field
- Client sends version with PATCH requests
- Server checks version matches before updating
- Returns 409 Conflict if mismatch
- Client shows "Card was edited by another user" message

SOFT DELETES (Audit Trail):
- Never hard delete from database
- Mark status as 'DELETED' or 'ARCHIVED'
- Preserve all history for audits
- Enable future restore functionality

CASCADE DELETES:
- DELETE /api/cards/[id] → Archives all associated benefits
- One atomic transaction (all-or-nothing)
- Rollback on any error

COMPONENT PATTERN (Radix UI):
- Use Dialog, DialogContent, DialogHeader
- React state for: form data, loading flag, errors, touched fields
- Try/catch with specific error handling for different status codes
- Toast notifications for user feedback
- Dark mode support (Tailwind CSS classes)
- Keyboard support (Escape to close)

CRITICAL PATTERNS
================================================================================

VALIDATION RULES:
- customName: String, max 100 chars, alphanumeric + spaces/dashes/parens
- actualAnnualFee: Integer >= 0, <= 1,000,000 (cents)
- renewalDate: ISO date string, must be future date
- benefitName: String, max 100 chars, unique per card
- userDeclaredValue: Integer >= 0, <= 10,000,000
- resetCadence: Enum (Monthly, CalendarYear, CardmemberYear, OneTime)

ERROR RESPONSES:
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "User-friendly message",
    "details": {
      "field": "fieldName",      // For validation errors
      "currentVersion": 2,       // For version conflicts
      "..." : "..."
    }
  }
}

EDGE CASES TO HANDLE:
- Version conflicts (two users editing same card)
- Duplicate benefit creation (rapid clicks)
- Network timeouts (30s limit with retry)
- Session expiration (redirect to login)
- Authorization failures (user tries to edit others' cards)
- Cascade delete failures (partial deletions rolled back)
- Form data preservation on error (sessionStorage)
