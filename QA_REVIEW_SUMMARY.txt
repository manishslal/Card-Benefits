================================================================================
AUTHENTICATION SYSTEM - QA REVIEW SUMMARY
================================================================================

Review Date: April 1, 2026
Status: CONDITIONAL APPROVAL - Needs critical fixes before Task #3

Overall Assessment: The authentication implementation is architecturally sound with
excellent security patterns (Argon2id, timing-safe comparison, JWT, AsyncLocalStorage).
However, 2 CRITICAL ISSUES prevent it from functioning in production and must be
fixed before proceeding to authorization implementation in Task #3.

================================================================================
CRITICAL BLOCKERS (Must Fix Before Task #3)
================================================================================

1. CRITICAL Issue #C1: AsyncLocalStorage Context Not Properly Scoped
   Location: /src/middleware.ts (lines 120-124)
   Impact: Server actions cannot access userId via getAuthUserId()
   
   The middleware extracts and validates the session but does NOT wrap the
   request handler with runWithAuthContext(). This means AsyncLocalStorage
   context is never established. When server actions call getAuthUserId(),
   they receive undefined.
   
   Fix Required: Use withAuth() wrapper in EVERY server action that needs userId.
   Estimated Fix Time: 2-3 hours for review + testing

2. CRITICAL Issue #C2: Signup Rate Limiting Not Implemented
   Location: /src/app/api/auth/signup/route.ts
   Impact: No protection against brute-force signup attacks
   
   Specification requires 3 signup attempts per hour per IP. No rate limiter
   is instantiated or used in signup route. Login route correctly implements
   rate limiting, but signup is wide open.
   
   Fix Required: Instantiate RateLimiter and check on every signup attempt
   Estimated Fix Time: 30 minutes

================================================================================
HIGH PRIORITY ISSUES (Should Fix Before Production)
================================================================================

3. HIGH Issue #H1: Middleware Cannot Wrap Async Context for Route Handlers
   Location: /src/middleware.ts
   Impact: API routes have no clean way to access userId
   
   Fix: Create shared cookie utility function + document pattern clearly

4. HIGH Issue #H2: Race Condition in Session Creation
   Location: /src/app/api/auth/signup/route.ts, login/route.ts
   Impact: Potential unique constraint violation on concurrent requests
   
   Fix: Generate token BEFORE creating session, not after

5. HIGH Issue #H4: Missing Database Index on Session.sessionToken
   Location: /prisma/schema.prisma
   Impact: Session lookups cause table scans on every request
   
   Fix: Add explicit @@index([sessionToken]) and @@index([isValid])

6. HIGH Issue #H5: Logout Returns 401 for Already Logged Out (Wrong HTTP Code)
   Location: /src/app/api/auth/logout/route.ts
   Impact: API semantics incorrect; 401 means "try again with credentials"
   
   Fix: Return 200 OK (already logged out = successful state)

================================================================================
MEDIUM PRIORITY ISSUES (Can Fix Next Iteration)
================================================================================

7. MEDIUM Issue #M1: Console.log Statements in Production Code
   Location: Multiple auth routes
   Impact: Error details leak in production logs

8. MEDIUM Issue #M2: Password Hash Length Validation Not Strict Enough
   Location: /src/lib/auth-utils.ts
   Impact: SESSION_SECRET entropy validation assumes 1 char = 1 byte

9. MEDIUM Issue #M3: Rate Limiter Memory Can Grow Unbounded
   Location: /src/lib/rate-limiter.ts
   Impact: Memory leak with many concurrent attackers

10. MEDIUM Issue #M4: No Validation That User Still Exists After Session Creation
    Location: signup/login routes
    Impact: Can create valid session for deleted user

11. MEDIUM Issue #M5: Session Creation Uses Empty String, Not Null
    Location: signup/login routes
    Impact: Confusing and error-prone; empty string is falsy in JS
